# 你的临床试验顾问

一个面向临床研发全生命周期的可复用 Agent Skill，适用于 Codex 及其他能够加载 Markdown 指令的AI工具。

它不是简单的法规搜索器。它会先明确用户真正需要作出的决定，再连接适用法规、ICH指导原则、中国NMPA/CDE监管要求、临床方法学和项目资料，形成可执行、可追溯的专业建议。

如果问题中的歧义会改变结论，它会先分轮追问到问题明确，再进入专业判断。用户还可以提供本人写过的文章或邮件，让顾问在保持事实准确的前提下，按其表达习惯回复或另写一版。

跨领域内容收录在《临床研究综合参考》中，适合全生命周期分析、培训和综合QC；高频判断与专题能力保留为精简reference，适合单点问题按需加载。两类资料分工不同，兼顾覆盖完整性和上下文效率。

## 能做什么

| 需求 | 默认产出 |
|---|---|
| 临床概念与术语解释 | 结论、适用边界、例子、常见误区和公开依据 |
| 官方指导原则检索 | 文件状态、适用性、正文定位和官方链接 |
| 法规与申报资料地图 | SUSAR/RSI、E2F、E3、E6、E9/E9(R1)、M4/CTD及相关CDE/NMPA/ICH文件路由 |
| 临床试验设计 | 临床问题、估计目标、人群、终点、对照、偏倚控制和可行性设计链 |
| 早期临床药理 | BA/BE、食物影响、FIH剂量递增、PopPK和DDI的证据—设计—决策框架 |
| 统计与估计目标 | 五属性估计目标、样本量参数框架、分析集、缺失、多重性和敏感性建议 |
| GCP、伦理与质量 | 受试者保护影响、质量风险、升级、记录、CAPA和有效性检查 |
| QA、稽查与检查准备 | 体系风险、稽查范围、根因、CAPA、有效性和持续检查准备 |
| 临床运营 | 可行性、中心、招募、供应商、风险导向监查、锁库和关闭控制点 |
| 数据与随机化 | CRF、数据流、电子系统、外部数据、随机化、盲态、对账和锁库控制 |
| 试验用药与资源 | 需求预测、温控、库存、清点、供应链、预算和变更控制 |
| 项目治理与医学监察 | RACI、关键路径、风险/问题/变更、决定日志和跨职能医学审核 |
| 注册策略 | 问题—证据规划、地区差异、监管互动准备和申报就绪检查 |
| 安全性与药物警戒 | AE/SAE/SAR/SUSAR分类、报告路径、信号和获益风险分析 |
| DSUR | DIBD/DLP/RSI锁定、资料包、章节依赖、缺口表和QC门 |
| CSR | 方案—SAP—数据库—TFL—正文—附录一致性链和修正建议 |
| 方法学QC | 总体判定、问题清单、风险优先级、修复方案和下一质量门 |
| 问题澄清 | 每轮1–3个高价值问题、问题画像和确认摘要 |
| 用户语气写作 | 从本人样本提取表达习惯，生成同语气回复、改写稿或新稿 |
| 用户本地记忆 | 经用户明确同意后，将稳定偏好或决定最小化写入用户侧MEMORY.md |

## 什么时候使用

当你需要回答以下问题时使用：

- 一个临床试验概念是什么、为什么重要、如何落地；
- 某个设计、终点、对照或统计策略是否合理；
- BA/BE、FIH剂量递增、PopPK或DDI研究应怎样建立决策框架；
- 样本量、非劣效、估计目标、分析集或缺失数据应如何处理；
- AE、SAE、SUSAR、方案偏离或安全信号应如何分类和处理；
- 中国或ICH是否存在适用的当前指导原则；
- 首份DSUR需要哪些资料，现有DSUR或CSR还缺什么；
- 现有方案、统计章节、资料包或工作流能否进入下一阶段；
- 中心、供应商、电子系统、试验用药、项目治理或检查准备是否形成闭环；
- 一个问题还比较模糊，需要通过分轮追问明确到可以作出可靠判断；
- 希望根据本人写过的文章、邮件或回复，生成一版符合自己表达习惯的新文本。
- 希望把长期偏好、项目约定或已确认决定保存到用户侧本地MEMORY.md。

## 工作方式

技能使用统一的证据—决策闭环：

`澄清到可判断 → 定义决策 → 建立问题画像 → 分配来源角色 → 官方检索与核验 → 方法学判断 → 行动化输出 → 质量核验`

它会区分法规要求、指导原则建议、方法学判断和项目实践建议；资料不足时列出缺口和影响，不用流畅文字填补事实。

## 语言与交付

- 默认跟随用户语言；中文用户默认获得清晰、专业的中文结果。
- 简单问题先给直接结论；复杂或高风险任务展开证据链、假设、限制和质量门。
- 用户需要成品时，交付可直接使用的检查表、资料需求表、判断树、章节框架、QC意见或正式文本。
- 问题不清楚且会改变结论时，不先给模糊答案；每轮只追问1–3个关键问题，直到达到可判断标准。
- 用户提供本人写作样本时，只迁移表达风格，不把旧稿中的日期、项目、人物、观点或承诺自动带入新稿。

## 证据与隐私边界

- 用户可见的规范性依据优先来自适用法规、ICH、NMPA、CDE等正式公开来源。
- 《临床研究问题点》和《临床研究综合参考》可以直接向用户展示，用于解释、培训、分析和工作底稿。
- 两份专业参考均不替代法规、官方指导原则或项目源文件。
- 法规不做成容易过期的静态全文库。涉及当前版本、时限、程序或强制义务时，由模型通过官方网页和原始附件实时核验。
- 当前要求、法定时限和产品特异性问题必须核验官方原文；无法核验时明确停止定论，不用本地知识代替当前官方依据。
- 无法可靠确认结论或官方依据时，不用模糊措辞误导用户；明确说明缺口，并提供适用官网、检索词和版本/状态/条款等核验字段，方便用户独立溯源或回传原文继续复核。
- 本 Skill 不替代监管机构正式沟通、伦理审查、统计师签字、研究者医学判断或法律意见。
- 如果Skill以公开仓库或本地Markdown形式分发，文件本身可以被安装者查看，因此不得在其中放入个人信息、受试者信息、未公开项目数据、私有文件路径或访问凭据。

## 用户侧本地记忆

- 默认不记录。只有用户明确提出“记住”或“保存到MEMORY.md”，且运行环境支持本地文件或记忆能力时才写入。
- 仅保存稳定偏好、已确认决定、项目术语和复核触发条件；首次写入前确认目标文件和范围。
- 不保存整段会话、患者或受试者信息、未公开项目数据、完整邮件、上传文件全文或reference全文。
- MEMORY.md只提供后续任务的上下文线索，不替代当前法规、项目文件或医学判断；用户可随时查看、更正、删除或停用。

## 安装到 Codex

将仓库目录复制到用户级 skills 目录：

```bash
mkdir -p "$HOME/.codex/skills"
cp -R clinical-trial-advisor "$HOME/.codex/skills/clinical-trial-advisor"
```

开启新任务后，技能可在临床试验相关请求中自动触发，也可以显式调用：

```text
$clinical-trial-advisor
```

示例：

```text
请帮我判断这个试验的主要终点和估计目标是否一致。
首份DSUR开写前必须锁定哪些资料？
这个给药前事件需要作为AE记录吗？
下面是我以前回复过的邮件，请按我的语气回复这次的新问题。
这个问题还没想清楚，请先逐步问我，明确后再给方案。
```

## 仓库结构

```text
clinical-trial-advisor/
├── SKILL.md                 # 唯一运行入口和工作流路由
├── agents/
│   └── openai.yaml          # Codex UI展示名称和默认提示词
├── scripts/
│   └── search_refs.py       # 专题知识检索工具
└── references/              # 1个综合reference + 5个专题reference
```

`SKILL.md`保持核心行为和路由；详细内容组织为6个reference：

- `临床研究综合参考.md`：跨领域综合参考，用于全生命周期分析、培训和综合QC；
- `clinical-research-questions.md`：临床研究问题点；
- `regulatory-guidance.md`：法规与指导原则工作参考，覆盖SUSAR/RSI、DSUR、CSR、E2F、E3、E6、E9/E9(R1)、M4/CTD、资料依赖、对账、QC及动态官网核验；
- `design-statistics.md`：试验设计与统计；
- `gcp-safety-operations.md`：GCP、安全与临床运营；
- `interaction-writing.md`：问题澄清、语气写作与用户本地记忆。

加载采用渐进式策略：单点问题优先读取最匹配的专题reference；跨职能综合问题、从0到1全链路分析或全面QC优先读取《临床研究综合参考》，再按缺口补充专题资料。综合reference与专题reference存在少量有意重叠，前者保证覆盖，后者降低窄问题的上下文成本。
